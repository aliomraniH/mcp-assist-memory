Connection layer

How clients connect

The transport and auth model we hardened so three surfaces — claude.ai, Claude Desktop and the Claude Code CLI — connect reliably and survive restarts.

🔑 Two managed tokens

One token per surface — web (token in the URL, for the headerless claude.ai connector) and desktop-cli (Bearer header, shared by Desktop + CLI). Each rotates independently from the password-gated /admin page.

♾️ Stateless transport

Every request is self-contained — no in-memory session affinity — so client sessions survive VM restarts & redeploys. This fixed the earlier intermittent "session" failures.

🛡️ Transparent retries

A dropped Postgres connection mid-call (Neon scale-down, 57P01) is retried on a fresh connection instead of erroring out to Claude — including the session-write path.

What makes it trustworthy

Cross-cutting guarantees

Properties that hold across every tool, not bolted onto any one of them.

  • Namespace = project = tenant. Every memory / handoff / session call is filtered by namespace — no implicit cross-project reads. A soft boundary today (any token holder can pass any namespace); per-project tokens are the v2 roadmap.
  • Append-only history. Writes never overwrite — every change is a new revision and deletes are tombstones, so the full trail is always recoverable.
  • Prompt-injection hardening. Values are sanitized on write and returned wrapped in <<<UNTRUSTED_DATA>>> markers — a defense against the "lethal trifecta." Use unwrap_value for the raw string.
  • Content-addressed artifacts. A blob's sha256 is its identity, so identical content is stored once and shared globally; the hash is the capability.
  • Idempotent writes. Pass a stable event_id for exactly-once saves during offline reconcile, so a retried write collapses to a no-op.
