# constraints.txt — deterministic, reproducible dependency pins for DEPLOY.
#
# WHY THIS EXISTS
#   pyproject.toml declares loose ranges (fastapi>=0.115, uvicorn>=0.30, ...).
#   Without a lock, every `pip install -e .` (dev, post-merge, and the deploy
#   build) is free to resolve NEWER versions than the ones actually verified in
#   dev. That is exactly how the prod /mcp 421 outage happened: an unpinned
#   fastmcp deploy pulled 3.4.3's HostOriginGuard. This file pins the ENTIRE
#   transitive closure so the deploy installs byte-for-byte what dev verified.
#
# HOW IT IS USED
#   `.replit`  build   : pip install -c constraints.txt -e .
#   post-merge.sh       : pip install -c constraints.txt -e .
#   A constraints file only *caps* versions of packages that actually get
#   installed; extra lines here are harmless. It never forces an install.
#
# HOW TO INTENTIONALLY UPGRADE A DEPENDENCY  (so these pins don't rot)
#   1. Loosen/adjust the range in pyproject.toml if needed.
#   2. Upgrade in dev:      pip install -U <pkg>       (or `-e .` to re-resolve)
#   3. Verify:              make test   AND exercise /mcp locally.
#   4. Regenerate pins:     pip freeze | grep -v mcp-assist-memory > constraints.txt
#      (then re-add this header, or run `scripts/lock-deps.sh`).
#   5. Redeploy — prod now installs the newly verified set.
#
# GENERATED FROM the verified-good dev environment. Do not hand-edit versions;
# regenerate via the steps above.
# ---------------------------------------------------------------------------
